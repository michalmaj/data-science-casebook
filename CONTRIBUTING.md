# Contributing

Thanks for considering a contribution to `data-science-casebook`. This document covers the conventions that aren't obvious from reading a single lesson.

## Local development

Install [uv](https://docs.astral.sh/uv/), then:

```bash
uv sync --locked --all-groups
```

Before pushing, reproduce what CI runs (`.github/workflows/ci.yml`), in order:

```bash
uv run --locked ruff check .
LESSON_MODULE=solution uv run --locked pytest
uv run --locked python tools/check_bilingual_pairs.py
```

`LESSON_MODULE=solution` tells every lesson's `check.py` to test `solution.py` (the reference answer) instead of `task.py` (the student scaffold, which is expected to fail with `NotImplementedError` until filled in) — this is how CI verifies the reference solutions stay correct, not how a student runs their own work.

## Adding a lesson

Every lesson lives in `cases/<case>/lessons/<NN_lesson_name>/` and needs exactly five files:

- `task.py` — the student-facing scaffold. Every function body is `raise NotImplementedError(...)`, with a docstring explaining what to implement and why.
- `solution.py` — the reference implementation. Same function signatures as `task.py`, real bodies.
- `check.py` — the self-check. See the module-loading pattern below.
- `lesson.ipynb` — the notebook the student actually works in, with cleared cell outputs (see "Notebook hygiene" below).
- `README.md` and `README.pl.md` — the lesson brief, in both languages.

**Self-containment convention**: if a lesson needs a function that an earlier lesson in the same case already defined (e.g. `load_dataset`, `split_dataset`), reproduce it byte-for-byte in the new lesson's `task.py`/`solution.py` — never import it from another lesson's module. Lessons must be runnable and gradable in isolation; a student jumping straight to Lesson 5 shouldn't need Lesson 3's files to exist. This means some duplication across a case's lessons is expected and intentional, not a bug to clean up.

**`check.py` module-loading pattern**: every `check.py` in this repo uses the same boilerplate to load either `task.py` or `solution.py` at runtime:

```python
import importlib.util
import os
from pathlib import Path

_MODULE_NAME = os.environ.get("LESSON_MODULE", "task")
_LESSON_DIR = Path(__file__).parent
_MODULE_PATH = _LESSON_DIR / f"{_MODULE_NAME}.py"
_UNIQUE_NAME = f"lesson_{_LESSON_DIR.parent.parent.name}_{_LESSON_DIR.name}_{_MODULE_NAME}"

_spec = importlib.util.spec_from_file_location(_UNIQUE_NAME, _MODULE_PATH)
lesson = importlib.util.module_from_spec(_spec)
_spec.loader.exec_module(lesson)
```

The `_UNIQUE_NAME` construction matters: many lessons across many cases have a file literally named `task.py`. Without a unique module name per lesson directory, Python's `sys.modules` cache would collide between them when pytest collects the whole repo in one run.

## Bilingual contract

Every tracked `*.md` file needs a `.pl.md` sibling (`tools/check_bilingual_pairs.py` enforces this in CI — it only considers files git would track, so this doesn't apply to gitignored files). English is the source of truth; when you edit an English Markdown file, update its Polish sibling in the same commit. The sole exemption is `do_poczytania.txt` (private, gitignored planning notes, not student-facing).

Code, docstrings, comments, and commit messages are English-only — including inside notebooks. `lesson.ipynb` markdown cells are English-only even in lessons whose `README.pl.md` is Polish; only the brief is bilingual, not the workspace.

## Regenerating a case's data

Each case's dataset is generated by `cases/<case>/data/generate.py`, using a fixed random seed — re-running it reproduces the exact same file byte-for-byte (CSV/SQLite) or semantically (Excel). If you change a case's data-generation logic (new column, different missingness pattern, different row count), re-run `generate.py` and then check every lesson's `check.py` in that case for reference values that assumed the old data — a data schema change is very likely to shift hardcoded numbers downstream.

## Notebook hygiene

Commit `lesson.ipynb` files with cleared outputs and no execution counts — a fresh notebook, not one you happened to run last. If you've been iterating in Jupyter, clear all outputs (Jupyter's "Restart Kernel and Clear All Outputs," or `jupyter nbconvert --clear-output --inplace lesson.ipynb`) before committing.

## Updating reference values in `check.py`

This repo's `check.py` files intentionally assert exact numeric values in most lessons (not just "the code runs without error") — that precision is what makes a lesson's self-check trustworthy. If a legitimate change to `solution.py` shifts a computed number (a bug fix, a data regeneration, a new feature added to a shared function), don't guess the new value or copy it from a single trusted-looking run: write a small throwaway script that independently recomputes it, confirm the number twice, and only then hardcode it into the assertion. A wrong hardcoded number that happens to match a buggy implementation is a silent correctness gap that's much harder to notice later than a test that simply fails.
